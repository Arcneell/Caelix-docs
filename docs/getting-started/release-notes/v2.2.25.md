## Backend — accès au magasin d'utilisateurs sortis de l'event loop (HIGH en cluster)

En mode cluster, le magasin d'utilisateurs vit dans etcd : chaque `get_user_by_username` /
`get_user_by_id` / `list_users` / `create_user` / `update_user` / `delete_user` est un aller-retour
réseau **bloquant** (`urllib` synchrone). Tous les handlers `async` de `routers/auth.py` les
appelaient directement sur l'event loop, gelant **toutes** les requêtes/SSE pendant chaque
aller-retour — `login` et `me` étant les chemins les plus chauds (toute tentative de connexion).

Chaque appel passe désormais par `asyncio.to_thread` : `login`, `me`, `change_password`
(3 appels), `get_users`, `create_new_user`, `update_existing_user` (3 appels), `delete_existing_user`
(2 appels). Le hashing bcrypt était déjà correctement déporté ; seuls les accès au magasin
restaient exposés. Avec ceci, la classe blocking-on-loop est **entièrement balayée** (routers +
dépendances auth + tâches de fond + middlewares).

## Cluster — lease de tombstone orphelin évité (LOW)

Suivi du correctif `remove_node` de 2.2.24 : sur le chemin d'erreur (retry après échec d'écriture
du tombstone), le lease etcd fraîchement accordé n'était pas révoqué → jusqu'à ~2 leases orphelins
(TTL 24 h, auto-expirants) par retrait raté. `remove_node` révoque désormais le lease avant de
réessayer.
