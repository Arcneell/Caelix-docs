# Notes de release — v2.2.18

**Date** : juillet 2026
**Image** : `ghcr.io/arcneell/caelix:2.2.18` (également `:latest`)

> Correctifs de suivi de l'audit : le tombstone de suppression (2.2.17) était effacé sous un
> id **non sanitisé** (nœud ré-ajouté bloqué), la boucle heartbeat rapide ne l'honorait pas,
> et deux `sleep` moteur restaient non gardés.

---

## Cluster — tombstone (suivi de 2.2.17)

- **Effacement au re-join corrigé.** L'agent écrit/lit le tombstone sous l'id **sanitisé**
  (`_node_sanitize_id`), mais `install.sh` l'effaçait sous l'id **brut** `--node-id` → un nœud
  ré-ajouté restait auto-fencé jusqu'à expiration du bail (24 h). `install.sh` applique
  désormais la même sanitisation.
- **Boucle heartbeat rapide.** Elle honore maintenant le tombstone (comme le fence) : un
  leader retiré ne re-pose plus la VIP pendant la fenêtre avant que la boucle principale ne le
  fence.

## Moteur

- `candidate_grace` / `post_repair_grace` (`lib/repair.sh`) passés à `sleep` sans garde → un
  `sleep infinity` aurait bloqué le chemin de réparation. Gardés `^[0-9]+$` (dernières valeurs
  manifeste non gardées).

## Frontend

- Deux libellés en dur (`Route`, `Template`) passés par `t()` (+ clés fr/en).

## Validation

Suite complète (747 pytest + 306 bats + vérifications shell + build frontend) au vert, puis
déploiement `2.2.18` sur le cluster 3 nœuds live (quorum etcd 3/3, VIP flottante, auth etcd
vérifiés ; les 3 nœuds conservent leur `/meta`, pas de faux fence).
