## Backend — last blocking-on-loop remnant in the Docker events SSE stream

`GET /api/docker/events/stream` (an `async` handler) resolved the target node via
`resolve_node_docker_host(node)` **directly** in the async body, before building the SSE
generator. In cluster mode that call reads the node registry from etcd (blocking `urllib`
round-trip) — on the event loop, once per connection to the stream. It is exactly the call that
`NodeTargetMiddleware` (main.py) already wraps in `asyncio.to_thread`; the streaming handler was
missed. Moved to `to_thread`.

For consistency, streaming subprocess teardown (`proc.terminate()`/`proc.wait()`, bounded to 5s,
in `finally`) is extracted into `_terminate_proc` and run via `to_thread` — no blocking
`subprocess` call remains even on teardown.

With this, the blocking-on-loop class is **fully closed**: an exhaustive re-sweep of the ~90
`async def` (route handlers, FastAPI dependencies, middlewares, background tasks, lifespan) shows
all blocking work (etcd store, subprocess, SMTP, bcrypt) is now under `to_thread` or in a sync
`def` (FastAPI threadpool).
