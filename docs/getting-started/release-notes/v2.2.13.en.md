# Release notes — v2.2.13

**Date**: July 2026
**Image**: `ghcr.io/arcneell/caelix:2.2.13` (also `:latest`)

> Fix: a container killed by the OOM killer is no longer mistaken for a manual operator
> stop.

---

## Fix — OOM kill ≠ manual stop

The engine suspends auto-restart of a container stopped "by hand" (`manual_stop_pause`),
keying on the exit code: `0`, `143` (SIGTERM) and `137` (SIGKILL) are read as an
intentional stop (`exit_code_looks_like_manual_stop`, `lib/repair.sh`).

But a container **killed by the OOM killer also exits 137**. It was therefore treated as a
manual stop → paused and **never restarted**, which could leave a service (including the
`caelix-ui` console) **down after every out-of-memory event**, until manual intervention.

Docker distinguishes the two via `.State.OOMKilled`. The manual-stop detection now
**excludes** OOM kills: an OOM-killed container falls through to the normal
restart/recreate path (like any failure), while a genuine operator stop is still honoured.

> Found during the VIP soak test (`scripts/vip_soak.sh`): a `docker kill` of the container
> (exit 137) left the console paused instead of being restarted.

## Validation

bats tests added (`repair_oom.bats`: 137+OOMKilled=true → restart; 137+OOMKilled=false →
pause). Full suite (pytest + bats + shell checks + frontend build) green, then the
`2.2.13` image was deployed to the live 3-node cluster with etcd quorum, floating VIP and
etcd auth verified.
