## Backend — dernier reliquat blocking-on-loop dans le flux SSE d'événements Docker

`GET /api/docker/events/stream` (handler `async`) résolvait le nœud cible via
`resolve_node_docker_host(node)` **directement** dans le corps async, avant de construire le
générateur SSE. En mode cluster, cet appel lit le registre des nœuds dans etcd (aller-retour
`urllib` bloquant) — sur l'event loop, à chaque ouverture de connexion au flux. C'est exactement
l'appel que `NodeTargetMiddleware` (main.py) enveloppe déjà dans `asyncio.to_thread` ; le handler
de streaming avait été manqué. Passé en `to_thread`.

Par cohérence, la fermeture des sous-processus de streaming (`proc.terminate()`/`proc.wait()`,
bornée à 5 s, en `finally`) est extraite dans `_terminate_proc` et exécutée via `to_thread` —
plus aucun appel `subprocess` bloquant même au démontage.

Avec ceci, la classe blocking-on-loop est **entièrement fermée** : re-balayage exhaustif des
~90 `async def` (handlers de route, dépendances FastAPI, middlewares, tâches de fond, lifespan),
tout le bloquant (store etcd, subprocess, SMTP, bcrypt) est désormais sous `to_thread` ou dans un
`def` sync (threadpool FastAPI).
