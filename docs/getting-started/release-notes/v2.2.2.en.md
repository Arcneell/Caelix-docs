# Release notes — v2.2.2

**Date**: July 2026
**Image**: `ghcr.io/arcneell/caelix:2.2.2` (also `:latest`)

> Security fixes from a **re-audit** of 2.2.1. Five fixes, two of which close gaps in
> 2.2.1 itself. No visible functional change.

---

## Fixes

- **Argument injection via `image` (container create)**: the `image` field is now
  validated (leading dash rejected) like the other fields, so a value such as
  `--entrypoint=…` can no longer be parsed as a `docker run` flag. Closes a gap in the
  2.2.1 hardening (the positional `image` field had been missed).
- **Join tokens repaired**: the console wrote the token under `caelix/join/tokens/` but
  the installer consumed it under `caelix/console/join/tokens/` (wrong key) — a
  legitimate join failed **and** the token was not actually single-use. Both sides now
  use the same key; the token is consumed atomically.
- **JWT refused in the `?token=` query**: a JWT in the URL leaks into proxy logs,
  history and `Referer` headers. Auth now accepts the token only from a header or
  cookie; EventSource uses the single-use `?ticket=` flow.
- **Login-lockout DoS fix**: the per-username lockout now blocks only **invalid**
  attempts — a valid password always goes through, so an attacker can no longer lock a
  known account (e.g. `admin`) out of the console. Per-IP and per-username brute-force
  protection is preserved.
- **etcd password off the command line at install**: creating the etcd user and the
  `etcdctl` calls no longer pass the password on argv (`ps` / `/proc/cmdline` leak) —
  user creation goes through the HTTP API (body on stdin) and auth uses the
  `ETCDCTL_USER` environment variable.

## Validation

Added regression tests (image injection, join-token key consistency, JWT-in-URL
refusal, valid login not blocked by the lockout). Full backend suite green; `etcdctl`
behaviour (user/add API, env auth) verified against a real etcd.
