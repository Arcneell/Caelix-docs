# Notes de release — v2.2.2

**Date** : juillet 2026
**Image** : `ghcr.io/arcneell/caelix:2.2.2` (également `:latest`)

> Correctifs de sécurité issus d'un **re-audit** de la 2.2.1. Cinq correctifs, dont
> deux comblent des lacunes de la 2.2.1 elle-même. Aucun changement fonctionnel visible.

---

## Corrections

- **Injection d'arguments via `image` (création de conteneur)** : le champ `image` est
  désormais validé (rejet du tiret initial) comme les autres champs. Une valeur du type
  `--entrypoint=…` ne peut plus être interprétée comme un flag `docker run`. Comble une
  lacune du durcissement 2.2.1 (le champ positionnel `image` avait été oublié).
- **Jetons de join réparés** : la console écrivait le jeton sous `caelix/join/tokens/`
  mais l'installeur le consommait sous `caelix/console/join/tokens/` (mauvaise clé) —
  un join légitime échouait **et** le jeton n'était pas réellement à usage unique. Les
  deux côtés utilisent maintenant la même clé ; le jeton est consommé atomiquement.
- **JWT refusé en query `?token=`** : un JWT dans l'URL fuit dans les logs de proxy,
  l'historique et les en-têtes `Referer`. L'authentification n'accepte plus le token que
  par en-tête ou cookie ; EventSource utilise le ticket à usage unique (`?ticket=`).
- **Anti-DoS sur le verrouillage de connexion** : le verrouillage par identifiant ne
  bloque plus que les tentatives **invalides** — un mot de passe valide passe toujours,
  donc un attaquant ne peut plus verrouiller un compte connu (ex. `admin`) hors de la
  console. La protection anti-force-brute par IP et par identifiant est préservée.
- **Mot de passe etcd hors ligne de commande à l'install** : la création de l'utilisateur
  etcd et les appels `etcdctl` ne passent plus le mot de passe en argument (fuite
  `ps`/`/proc/cmdline`) — création via l'API HTTP (corps sur stdin) et authentification
  via la variable d'environnement `ETCDCTL_USER`.

## Validation

Tests de non-régression ajoutés (injection `image`, cohérence de clé du jeton de join,
refus du JWT en URL, login valide non bloqué par le lockout). Suite backend complète
verte ; comportement `etcdctl` (API user/add, env auth) vérifié contre un etcd réel.
