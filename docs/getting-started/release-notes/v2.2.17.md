## Cluster — tombstone de suppression (H3)

`remove_node` supprimait les clés d'un nœud mais **sans tombstone**. Un nœud
partitionné-mais-vivant (ou retiré alors qu'il était sain) **re-publiait** ses clés
(`meta`/`heartbeat`/backends) à sa passe d'agent suivante et **réapparaissait** dans le
cluster : le scheduler le re-ciblait, l'ingress repointait dessus → double-exécution et
flapping. Le self-fence promis par la doc n'existait pas.

Désormais :
- `remove_node` écrit un **tombstone leasé** `caelix/nodes/<id>/tombstone` (après suppression
  des clés en **un seul `deleterange` atomique** — corrige aussi la non-atomicité) ;
- l'agent lit le tombstone en tête de `node_agent_cycle` : s'il est présent → **self-fence**,
  relâche la VIP et ne publie plus rien ;
- un **re-join autorisé** (gaté par le join-token à usage unique) efface le tombstone, si bien
  qu'un nœud retiré peut être ré-ajouté proprement ; le bail fait aussi expirer le tombstone
  comme filet de sécurité.

## Validation

Tests : suppression → tombstone présent + clés parties + re-join efface (FakeEtcd) ; agent
self-fence sous tombstone / passe normale sinon (bats). Suite complète (747 pytest + 306 bats
+ vérifications shell + build frontend) au vert, puis déploiement `2.2.17` sur le cluster
3 nœuds live (quorum etcd 3/3, VIP flottante, auth etcd vérifiés).
