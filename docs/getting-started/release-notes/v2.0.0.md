# Notes de release — v2.0.0

**Date** : juin 2026  
**Image** : `ghcr.io/arcneell/caelix:2.0.0` (également `:latest`)

> **2.0** rend le **cluster hautement disponible** prêt à l'emploi ET apporte une
> **console entièrement repensée** : moderne, à plat (style NetBird/Portainer) et
> cluster-native. Validée de bout en bout sur un cluster 3 nœuds (console, ingress
> VIP, état partagé, HPA, bascule) et en mono-hôte.

---

## Console repensée (cluster-first)

- **Navigation à plat, ~8 sections** (Vue d'ensemble · Nœuds · Conteneurs · Services ·
  Stacks · Ingress · Activité · Réglages) avec des **onglets** par section — fini les
  sous-menus imbriqués hérités du mono-hôte.
- **Vue d'ensemble = topologie** : cartes des nœuds (rôle/leader/VIP, santé, CPU·RAM,
  compteurs ressources par nœud), KPIs cluster, quorum, incidents récents.
- **Section Nœuds** dédiée (liste + drain/undrain + détail par nœud).
- **Tout est agrégé multi-nœuds** : conteneurs/images/volumes/réseaux montrent une
  colonne **Nœud** et un filtre ; chaque action vise le bon nœud. **Notifications**
  agrégées de tous les nœuds ; **événements** ciblables par nœud.
- **Rendu progressif** (un nœud lent ne bloque plus la liste) et **virtualisation** des
  longues listes. **Mono-hôte** : la console se simplifie automatiquement (pas de
  section Nœuds, pas de colonne Nœud). FR/EN, clair/sombre.

## Résumé

Le mode **cluster devient hautement disponible par conception** ET se gère **comme un
mono-hôte** depuis la console : tous les nœuds sont membres etcd + controllers, le
maillage WireGuard est obligatoire, une **VIP flottante** donne une adresse stable qui
**bascule automatiquement**, l'état de la console est **partagé** entre les nœuds, on
**pilote le Docker de n'importe quel nœud** depuis l'UI, et un **HPA** ajuste le nombre
de réplicas selon la charge.

## Cluster « comme en mono-hôte »

- **Console cohérente en HA** : utilisateurs, secret JWT et **tout l'état console**
  (registries, domaines, certificats + **PEM TLS**, webhooks, app-stacks, templates de
  service/custom, **stacks Docker Compose** et leurs `.env`) sont **partagés via etcd**
  → login, sessions, config, stacks et certificats identiques sur tous les nœuds et
  après une bascule. Les contenus liés au disque (compose, PEM) sont **matérialisés**
  localement avant usage (`docker compose`, proxy TLS). Compose v2 est inclus dans
  l'image pour déployer les stacks depuis la console.
- **Ciblage Docker par nœud** : la console pilote conteneurs/images/volumes/réseaux/
  stacks/logs de **n'importe quel nœud** via l'en-tête `X-Caelix-Node` (dockerd exposé en
  TCP, restreint au réseau interne ; mTLS pour le durcissement prod).
- **HPA (autoscale horizontal)** : `hpa = 1` + `hpa_min`/`hpa_max`/`hpa_target` (CPU %)
  /`hpa_cooldown` — le leader ajuste `total_replicas` selon le CPU des réplicas, le
  scheduler place et l'ingress load-balance.

## Nouveautés

- **Cluster HA automatique** — chaque nœud est un **membre etcd** (quorum Raft) et un
  **controller** (`CAELIX_CONTROLLER=1`). Le bail etcd + la transaction put-if-absent
  élisent un seul leader actif ;
  avec **≥ 3 nœuds**, la perte du leader est absorbée (quorum 2/3) et un autre nœud
  reprend le leadership + la VIP en quelques secondes. Le nœud d'amorçage
  (`--mode controller`) bootstrappe le raft, les autres le rejoignent
  (`--mode join --store-addr …`).
- **Maillage WireGuard obligatoire** — un nœud cluster ne peut exister sans : l'installeur
  exige `wireguard-tools`, et l'agent **applique le maillage automatiquement** à chaque
  cycle (clés générées au 1er cycle, directives publiées par le leader).
- **Install cluster autonome** — `install.sh` met en place tous les prérequis (Docker,
  socat, wireguard-tools + `modprobe`, arping pour la bascule VIP). Nouveau flag
  `--admin-password` (même valeur sur tous les nœuds → admin cohérent après bascule ;
  sinon un mot de passe fort est généré et affiché une fois dans les logs de la console).
- **VIP de cluster** — activée par `CAELIX_CLUSTER_VIP` (`--mode controller --vip
  10.0.0.10/32`). L'agent du **leader** pose la VIP (`CAELIX_VIP_IFACE` sinon route par
  défaut) + ARP gratuit ; les autres la relâchent. Console (`:18100`) et ingress (`:80`),
  qui écoutent en `0.0.0.0`, répondent sur la VIP. La VIP est **publiée dans le store**
  pour qu'un nœud promu leader la reprenne sans reconfiguration. Diagnostic :
  `caelix vip-status`. Flag `--cluster-size N` (défaut 3).

## Corrections

- **Service cluster joignable via la VIP** — deux défauts empêchaient un service
  cluster déployé depuis la console (image + `publish` + `autoscale_route`) d'être
  servi par l'ingress :
  - le moteur de réparation défaut `health_type` à `http`, donc un service sans
    sonde explicite était jugé `health_url_manquant` puis réparé/purgé/suspendu
    (le conteneur n'était jamais (re)créé avec son `-p`). Le splitter de manifeste
    cluster injecte désormais `health_type = none` par défaut (une sonde explicite
    reste prioritaire) ; la reprise au crash passe toujours par `create_missing`.
  - le hash de config du proxy global incluait `routes.conf` — le fichier que la
    mise à jour des routes produit — et la mise à jour était sautée quand le hash
    correspondait (boucle). `routes.conf` est maintenant régénéré à chaque passe
    (le handler le relit par connexion, aucun redémarrage), le hash ne gouverne que
    le redémarrage du listener.
- **Proxy / verrou global** : les processus longue-durée (proxy global et load-balancer
  d'autoscale) n'héritent plus du descripteur du verrou global (`flock`). Auparavant un
  `socat` pouvait conserver le verrou ouvert et **bloquer toutes les passes de
  réconciliation suivantes**. Le fd est désormais fermé au lancement (`200>&-`).

## Sécurité

- **dockerd TCP et etcd liés à l'IP privée du nœud** — l'install ne lie plus
  `dockerd:2375` ni l'API etcd `:2379` à `0.0.0.0` mais à l'**IP de cluster** du nœud.
  2375 est non authentifié (= RCE root), il ne doit jamais écouter sur une interface
  publique ; le binding privé est la vraie barrière, le pare-feu reste en défense en
  profondeur.
- **`docker_addr` validé avant usage** — le point de terminaison Docker d'un nœud
  (ciblage `X-Caelix-Node`) lu depuis le store est désormais validé (`tcp://hôte:port`
  ou `unix://`) avant de devenir `DOCKER_HOST`, fermant un vecteur SSRF si le store est
  empoisonné.
- **Matérialisation durcie** — l'écriture sur disque des certs/stacks/templates partagés
  rejette tout nom contenant un séparateur ou `..` (anti-traversée), et les fichiers
  compose sont écrits atomiquement (tmp + `rename`).
- **Prérequis de durcissement cluster** (à faire côté opérateur) : le KV etcd est le
  plan de contrôle et contient le secret JWT, les hash de mots de passe et les clés TLS.
  En production, **activez le TLS** sur l'API etcd (client et pairs). Sans TLS,
  quiconque atteint l'API etcd du cluster peut lire ces secrets.
