# Notes de release — v2.0.1

**Date** : juin 2026  
**Image** : `ghcr.io/arcneell/caelix:2.0.1` (également `:latest`)

> Release de **correctifs** sur la 2.0, issue d'une campagne de tests de bout en bout
> sur un cluster 3 nœuds réel et en mono-hôte. Elle rend les opérations cluster
> (réduction d'échelle, suppression de service, bascule) **convergentes et fiables**,
> et corrige plusieurs points mono-hôte. Aucune rupture de compatibilité.

---

## Corrections cluster (les plus importantes)

- **La réduction d'échelle et la suppression convergent enfin.** Un nœud retiré du plan
  conservait son ancien sous-manifeste et ne supprimait jamais les répliques
  enlevées/réduites. L'agent adopte désormais aussi un sous-manifeste **présent mais
  vide** (seules les sections épinglées localement — console, proxy — survivent), si
  bien qu'un `total_replicas` réduit, un HPA qui descend ou la suppression d'un service
  **se propagent et se nettoient tout seuls**, sans redémarrage d'agent.
- **Plus de backends ingress fantômes.** Le backend d'une réplique supprimée restait
  enregistré et la VIP:80 continuait de router vers une réplique disparue. L'agent
  **dé-enregistre** maintenant les backends des apps absentes de son manifeste.
- **Plus de conflit de port en bascule.** Un service ingress à **port hôte fixe** est
  désormais plafonné à **une réplique par nœud** : à la perte d'un nœud, la réplique en
  trop passe *en attente* au lieu de boucler sur `port is already allocated`.
- **`remove_orphans` actif par défaut sur chaque nœud du cluster** : le plan du
  controller fait autorité, donc tout conteneur `caelix-*` hors plan est nettoyé (une
  valeur explicite dans `[orchestrator]` reste prioritaire).

## Corrections mono-hôte

- **Sauvegarde de volume depuis la console** : un bind-mount sous la racine Caelix
  échouait (`no_existing_paths`) car le backend testait le chemin **hôte** depuis le
  conteneur. Les chemins sont maintenant traduits vers leur emplacement dans le
  conteneur avant l'archivage.
- **Mot de passe admin initial** : il est désormais aussi écrit dans
  `.caelix/initial-admin-password` (0600), affiché par l'installeur, et supprimé au
  premier changement de mot de passe — il n'est plus perdu si le conteneur console est
  recréé pendant l'installation.
- **`caelix status`** ne se fie plus au répertoire courant (`./.caelix`) : il lit l'état
  depuis la racine d'installation, donc plus de faux « démon inactif ».
- **Documentation** : la connexion par défaut décrite (`admin`/`admin`) est corrigée —
  le mot de passe est **aléatoire** au premier démarrage.

## Mise à jour

```bash
docker run --rm ghcr.io/arcneell/caelix:latest cat /opt/caelix/install.sh | bash -s -- --with-systemd
```

En cluster, mettez à jour chaque nœud (image + ré-extraction du moteur) puis laissez
l'agent réconcilier. Aucune modification de configuration n'est requise.
