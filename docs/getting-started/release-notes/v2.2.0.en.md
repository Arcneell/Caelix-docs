# Release notes — v2.2.0

**Date**: July 2026  
**Image**: `ghcr.io/arcneell/caelix:2.2.0` (also `:latest`)

> **Full cluster-aware** release. Every operational path — deploy, service/container
> actions, logs, manifest-derived views, backups, manual scale, compose stacks,
> template deploy, ingress, Let's Encrypt certificates and the floating VIP — now
> resolves the hosting node, reads the cluster manifest, or replicates its state via
> etcd, and **survives a VIP failover**. Single-host behaviour is preserved throughout.

---

## Cluster-aware

- **Let's Encrypt certificates** replicated across nodes: ingress and TLS follow the
  VIP. The proxy forwards `X-Forwarded-Proto` to apps behind TLS (fixes missing CSS/JS
  over HTTPS, e.g. WordPress).
- **Service/container actions, logs and views** target the node that actually hosts the
  container (previously the local daemon only — a silent no-op when the app ran
  elsewhere).
- **Backups** run on the node that owns the data; `status`/`list`/`download` aggregate
  archives across all three nodes.
- **Manual scale** via `total_replicas` in the cluster manifest (the controller places
  replicas across nodes).
- **Compose stacks** pinned to their deploy node and controllable from any node; `list`
  aggregates status across every node.
- **VIP failover** is clean on graceful agent stop (drain via `ExecStop`); replicated
  domain routes and certificates let the new leader serve everything immediately.

A reusable helper (`core/cluster/aware.py`) centralises node resolution, cluster-manifest
merging and multi-node fan-out.

---

## Validation

Tested live on a 3-node cluster, **including a full VIP failover**: HTTPS ingress,
cross-node actions, aggregated backup status and compose-stack control all keep working
on the new leader.
