## Security — per-app backup credentials hidden from technicians (HIGH)

`GET /api/state` returned, for the manifest's (non-reserved) app sections, **every** field except
`env`/`volumes_bind`. But `lib/backup.sh` reads backup credentials straight off the app section
as top-level keys (`backup_ftp_pass`, `backup_rsync_pass`, `backup_rsync_ssh_key`) — none of which
belonged to `_NOTIFY_REDACT_KEYS`. So a **technician** received these secrets in cleartext via
`/state`, contradicting the boundary already enforced on `notify` and reserved sections.

`sanitize_state_for_role` now applies a secret-key predicate (`_is_secret_manifest_key`) to **all**
manifest sections (reserved and app alike): it covers `_NOTIFY_REDACT_KEYS`, generic names
(`password`, `token`, `secret`, `api_key`, `private_key`, `credential`) and conventional suffixes
(`*_pass`, `*_password`, `*_secret`, `*_token`, `*_api_key`, `*_ssh_key`, `*_private_key`) — so
future secret fields are covered too. The technician-no-secrets boundary is now closed across every
shape of `/state`.
