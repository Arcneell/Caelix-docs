# Release notes — v2.2.24

**Date**: July 2026
**Image**: `ghcr.io/arcneell/caelix:2.2.24` (also `:latest`)

> The post-2.2.23 confirmation audit did **not** confirm convergence: it reopened the
> technician-no-secrets class (one root sanitizer hole + 5 endpoints returning raw state) plus
> 3 blocking I/Os and 2 engine/cluster defects. All fixed here.

---

## Security — secret leaks to technicians (HIGH)

**Root hole: `state["apps"]` was never sanitized.** `collect_state` builds the `apps` list by
copying **every** raw key of each app section (`app.update(values)`) — so `env`, `volumes_bind`
and the backup credentials. `sanitize_state_for_role` only rewrote `notify` and `manifest`,
never `apps`: a technician read the secrets via `state.apps[i]` even though `state.manifest` was
clean. The "safe" `GET /state` and the SSE `/stream` therefore leaked anyway. The sanitizer now
scrubs `apps[]` identically.

**5 `require_auth` endpoints returned raw `collect_state()`** (skipping the sanitizer that
`GET /state` applies): `POST /alerts/ack`, `POST /alerts/ack_all`, `POST /reconcile-app`,
`POST /service/action` (action `start`, technician-reachable) and `POST /container/action`
(same). Each now sanitizes the returned state by the caller's role, via the new
`sanitize_state_for_request` helper.

**`GET /api/volumes/{name}/file` is now admin-only.** It returned up to 1 MB of raw content of
any volume file to any authenticated user — yet volumes routinely hold `.env`/creds. Now
`require_admin`, matching `manifest/raw` and `inspect` (Env).

## Backend — 3 blocking I/Os moved off the event loop

- **`get_current_user`** (auth dependency on **every authenticated request**) called
  `get_user_by_username` synchronously → in cluster mode a blocking etcd read per request.
  Moved to `asyncio.to_thread`. This was the hottest, most critical path.
- **`_cert_sync_loop`** (background task, every 60 s) called `materialize_certs` /
  `materialize_domain_routes` directly (etcd + disk) on the loop → froze the whole service.
  Moved to `to_thread`.
- **`POST /templates/sources`** wrote the shared store (etcd) directly in an `async` handler.
  Moved to `to_thread`.

## Engine & cluster (MED)

- **Ingress proxy busy-loop**: `[proxy].health_interval` / `connect_timeout` were forwarded as
  env vars to `proxy.sh`, bypassing its CLI numeric guard. A non-integer value (e.g. `3s`) made
  `sleep` fail every tick → health-check loop at 100% CPU on the ingress node. Numeric coercion
  added in `autoscale_proxy.sh`.
- **Silent node resurrection**: if the tombstone write failed in `remove_node` (etcd blip right
  after the key delete), the error was only logged — the node came back on its next re-join while
  the API reported a successful removal. `remove_node` now retries then **raises**; the
  `DELETE /nodes/{id}` handler surfaces it (500) so the admin retries (removal is idempotent).

## Considered, left as-is
Webhook token readable by a technician (**adjudicated by-design** in 2.2.20: encoded by a test);
registry `username` (no password is ever persisted); template source URLs.

## Validation
Full suite green (**1000 pytest** + 306 bats + check-all + black + flake8), `apps[]` and endpoint
leaks reproduced in tests then closed, `2.2.24` deployed to the live 3-node cluster.
