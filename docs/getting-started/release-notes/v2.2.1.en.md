# Release notes — v2.2.1

**Date**: July 2026
**Image**: `ghcr.io/arcneell/caelix:2.2.1` (also `:latest`)

> **Security hardening** release. A full code audit (auth, Docker execution, cluster
> control plane, secrets, install scripts) followed by fixing every finding. Functional
> behaviour is unchanged; the new cluster-side security features are **backward
> compatible** (only active when configured).

---

## Cluster control plane (etcd)

- **etcd client authentication**: the console and the agent authenticate to etcd
  (`/v3/auth/authenticate`, token refreshed on 401). The installer enables auth
  idempotently at bootstrap (on by default in cluster mode; `--no-etcd-auth` to opt
  out). Mere network reachability to `:2379` no longer reads the KV store.
- **At-rest secret encryption**: the JWT signing secret, user accounts and TLS private
  keys are encrypted (Fernet/AES) before being stored in etcd. The master key
  (`CAELIX_MASTER_KEY`) is **never** stored in etcd — it is distributed to each node via
  the 0600 cluster env file. Legacy plaintext is read transparently and re-encrypted (a
  one-time boot migration).
- **Single-use join tokens**: `Cluster → Add a node` issues a TTL-bound token, consumed
  atomically on first use — a join command can no longer be replayed.

## Auth & access control

- Constant-time comparison of the legacy `CAELIX_UI_TOKEN`.
- The role is re-read from the database on every request (a demotion/deactivation takes
  effect immediately instead of waiting for the JWT to expire).
- **Per-username** login lockout (global cap) on top of per-IP: an attacker rotating
  source IPs can no longer brute-force a single account.
- `/metrics` requires authentication **by default** (`CAELIX_METRICS_PROTECT=0` to
  expose it to an internal Prometheus scrape on a trusted network).
- `exec` is forbidden inside Caelix system containers (console/etcd) and audit-logged.

## Docker execution & ingress

- Container creation rejects dash-prefixed option values (argument injection) and
  `env_file` paths with traversal.
- The autoscaler (HPA) validates `docker_addr` before using it as `DOCKER_HOST` (closes
  an SSRF → RCE vector on the controller).
- Ingress backends read from the registry are validated (`host:port`) before feeding
  the proxy — a poisoned entry is dropped.

## Secrets & scripts

- Secret files (JWT secret, TLS keys, `notify.ini`) written 0600 **from creation** (no
  world-readable window).
- The installer chmods `notify.ini` / `manifest.ini` to 0600 (SMTP passwords, bot
  tokens, webhooks, backup credentials).
- SMTP/FTP credentials passed via a 0600 `curl --config` file instead of the command
  line (argv is world-readable in `/proc`).
- Validation of rsync SSH key / known_hosts config (blocks `ssh -o ProxyCommand=…`
  option injection).

---

## Validation

Audit findings fixed and verified. Full backend suite green (701 tests); etcd auth,
at-rest encryption and join tokens validated against a real authenticated etcd. The
3-node cluster was redeployed to 2.2.1, at-rest secret encryption enabled and migrated,
with quorum and the VIP preserved.
