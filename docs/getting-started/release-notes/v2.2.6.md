## Convention de réponse unifiée

Les endpoints construisaient leurs réponses d'erreur de plusieurs façons (JSONResponse
`{ok:false, error}`, `HTTPException` renvoyant `{detail}`, tuples `(code, dict)`). Deux
helpers canoniques les unifient :

- **`json_ok(**champs)`** → `{"ok": true, ...}`
- **`json_err(message, status=400)`** → `{"ok": false, "error": message, "detail": message}`

`json_err` renvoie **à la fois** `error` et `detail`, donc le frontend (qui lit `detail`
pour 401/403/429 et `detail || error` ailleurs) fonctionne à l'identique — **zéro
changement côté client**.

- ~130 réponses `JSONResponse({ok:false, error}, status)` ralliées à `json_err`.
- Les 18 endpoints de `auth.py` passent d'`HTTPException` à `json_err` (login, changement
  de mot de passe, gestion des utilisateurs) — mêmes codes de statut, même `detail`.
- Les **dépendances** et helpers d'abort (`require_auth`, `_require_store`…) **continuent**
  de lever `HTTPException` : une dépendance FastAPI ne peut pas retourner de réponse —
  c'est la forme correcte, et le client la gère déjà.

## Correctif

- **`POST /api/autoscale/stress`** renvoyait ses **erreurs en HTTP 200** (dict brut
  retourné directement) — le client ne pouvait pas les détecter. Elles sortent désormais
  avec le bon statut (500).

## Validation

Suite backend complète verte (**712 + helpers**) ; contrat d'erreur d'auth vérifié
(status + `detail` préservés). Console re-buildée et servie sur le cluster 3 nœuds.
