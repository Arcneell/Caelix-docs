# Release notes — v2.2.6

**Date**: July 2026
**Image**: `ghcr.io/arcneell/caelix:2.2.6` (also `:latest`)

> Unifies the API routers' response convention + one status-code fix. No client-visible
> change (the wire contract is preserved).

---

## Unified response convention

Endpoints built their error responses several ways (JSONResponse `{ok:false, error}`,
`HTTPException` returning `{detail}`, `(code, dict)` tuples). Two canonical helpers unify
them:

- **`json_ok(**fields)`** → `{"ok": true, ...}`
- **`json_err(message, status=400)`** → `{"ok": false, "error": message, "detail": message}`

`json_err` returns **both** `error` and `detail`, so the frontend (which reads `detail`
for 401/403/429 and `detail || error` elsewhere) works identically — **zero client
change**.

- ~130 `JSONResponse({ok:false, error}, status)` responses moved to `json_err`.
- The 18 `auth.py` endpoints moved from `HTTPException` to `json_err` (login, password
  change, user management) — same status codes, same `detail`.
- **Dependencies** and abort-helpers (`require_auth`, `_require_store`…) **keep** raising
  `HTTPException`: a FastAPI dependency cannot return a response — that is the correct
  form, and the client already handles it.

## Fix

- **`POST /api/autoscale/stress`** returned its **errors as HTTP 200** (the raw dict was
  returned directly), so the client couldn't detect them. They now go out with the
  correct status (500).

## Validation

Full backend suite green (**712 + helpers**); auth error contract verified (status +
`detail` preserved). Console rebuilt and served on the 3-node cluster.
