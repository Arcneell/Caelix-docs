# Notes de release — v2.2.8

**Date** : juillet 2026
**Image** : `ghcr.io/arcneell/caelix:2.2.8` (également `:latest`)

> Correctif : les sauvegardes de bases **MySQL** échouaient silencieusement.

---

## Correctif — sauvegarde DB sur images MySQL

La sauvegarde logique de base de données (`backup_db_type = mysql|mariadb`) utilisait
`mariadb-dump` pour les deux moteurs. Or les images **`mysql:*`** ne fournissent que
`mysqldump` (pas `mariadb-dump`), donc le dump échouait (`command not found`).

Avant le durcissement 2.2.3, cet échec était **masqué** (le gzip d'une sortie vide
produisait une archive non nulle → enregistrée comme « réussie »). Depuis 2.2.3
(vérification `PIPESTATUS` + `gzip -t`), l'échec réel est correctement détecté — mais
comme un backup en échec ne met pas à jour son horodatage, il était **retenté à chaque
passe de l'agent**, floodant les incidents (`backup_failed` en boucle).

Le backup choisit désormais l'outil présent dans le conteneur : `mariadb-dump` si
disponible (images MariaDB), sinon `mysqldump` (images MySQL, et alias de compat
MariaDB). Les sauvegardes MySQL produisent à nouveau un dump valide.

> Si vous voyiez des incidents `Échec sauvegarde` en boucle sur une app à base MySQL,
> ils cessent avec cette version (le backup réussit et enregistre `backup_completed`).

## Validation

Vérifié en direct sur le cluster 3 nœuds : le dump `wordpress-db` (image `mysql:8`)
réussit (~840 Ko gzip, `gzip -t` OK), l'incident passe de `backup_failed` à
`backup_completed`, et le retry en boucle s'arrête.
