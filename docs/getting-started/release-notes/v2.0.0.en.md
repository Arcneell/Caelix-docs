# Release notes — v2.0.0

**Date**: June 2026  
**Image**: `ghcr.io/arcneell/caelix:2.0.0` (also `:latest`)

> **2.0** makes the **highly-available cluster** ready to use AND ships a **fully
> redesigned console**: modern, flat (NetBird/Portainer style) and cluster-native.
> Validated end-to-end on a 3-node cluster (console, VIP ingress, shared state, HPA,
> failover) and on single-host.

---

## Redesigned console (cluster-first)

- **Flat navigation, ~8 sections** (Overview · Nodes · Containers · Services · Stacks ·
  Ingress · Activity · Settings) with **tabs** per section — no more nested submenus
  inherited from single-host.
- **Overview = topology**: node cards (role/leader/VIP, health, CPU·RAM, per-node
  resource counts), cluster KPIs, quorum, recent incidents.
- A dedicated **Nodes** section (list + drain/undrain + per-node detail).
- **Everything aggregates across nodes**: containers/images/volumes/networks show a
  **Node** column + filter, and every action targets the right node. **Notifications**
  aggregate from all nodes; **events** can target a node.
- **Progressive rendering** (a slow node no longer blocks the list) and **virtualized**
  long lists. **Single-host** simplifies automatically (no Nodes section, no Node
  column). FR/EN, light/dark.

## Summary

Cluster mode is now **highly available by design** AND is managed **like a single
host** from the console: every node is an etcd member + controller, the WireGuard mesh
is mandatory, a **floating VIP** gives a stable address that **fails over
automatically**, the console state is **shared** across nodes, you **drive any node's
Docker** from the UI, and an **HPA** scales replicas with load.

## Cluster "like a single host"

- **HA-coherent console**: users, the JWT secret and **all console state** (registries,
  domains, certificates + **TLS PEMs**, webhooks, app-stacks, service/custom templates,
  **Docker Compose stacks** and their `.env`) are **shared via etcd** → login,
  sessions, config, stacks and certificates are identical on every node and after a
  failover. Disk-bound content (compose, PEMs) is **materialized** locally before use
  (`docker compose`, the TLS proxy). Compose v2 ships in the image so stacks deploy
  from the console.
- **Per-node Docker targeting**: the console drives containers/images/volumes/networks/
  stacks/logs of **any node** via the `X-Caelix-Node` header (dockerd exposed over TCP,
  restricted to the internal network; mTLS for production hardening).
- **HPA (horizontal autoscaler)**: `hpa = 1` + `hpa_min`/`hpa_max`/`hpa_target` (CPU %)
  /`hpa_cooldown` — the leader adjusts `total_replicas` from replica CPU, the scheduler
  places and the ingress load-balances.

## New

- **Automatic cluster HA** — every node is an **etcd member** (Raft quorum) and a
  **controller** (`CAELIX_CONTROLLER=1`). The etcd lease + put-if-absent transaction
  elects a single active leader;
  with **≥ 3 nodes**, losing the leader is absorbed (2/3 quorum) and another node takes
  over leadership + the VIP within seconds. The bootstrap node (`--mode controller`)
  bootstraps the raft, the others join it (`--mode join --store-addr …`).
- **Mandatory WireGuard mesh** — a cluster node cannot exist without it: the installer
  requires `wireguard-tools`, and the agent **applies the mesh automatically** every
  cycle (keys generated on the first cycle, directives published by the leader).
- **Self-contained cluster install** — `install.sh` sets up every prerequisite (Docker,
  socat, wireguard-tools + `modprobe`, arping for VIP failover). New `--admin-password`
  flag (same value on every node → consistent admin after failover; otherwise a strong
  password is generated and printed once in the console logs).
- **Cluster VIP** — enabled with `CAELIX_CLUSTER_VIP` (`--mode controller --vip
  10.0.0.10/32`). The **leader**'s agent adds the VIP (`CAELIX_VIP_IFACE` else the
  default-route interface) + gratuitous ARP; others release it. Console (`:18100`) and
  ingress (`:80`), listening on `0.0.0.0`, answer on the VIP. The VIP is **published to
  the store** so a promoted node takes it over without reconfiguration. Diagnostic:
  `caelix vip-status`. Flag `--cluster-size N` (default 3).

## Fixes

- **Cluster service reachable through the VIP** — two defects kept a cluster service
  deployed from the console (image + `publish` + `autoscale_route`) from being served
  by the ingress:
  - the repair engine defaults `health_type` to `http`, so a service with no explicit
    probe was flagged `health_url_manquant` then repaired/purged/suspended (the
    container was never (re)created with its `-p`). The cluster manifest splitter now
    defaults `health_type = none` (an explicit probe always wins); crash recovery still
    fires via `create_missing`.
  - the global proxy config hash included `routes.conf` — the file the route update
    produces — and the update was skipped when the hash matched (circular). routes.conf
    is now regenerated every pass (the handler re-reads it per connection, no restart);
    the hash covers only the listener and gates only a restart.
- **Proxy / global lock**: long-lived processes (the global proxy and the autoscale
  load-balancer) no longer inherit the global lock file descriptor (`flock`). Previously
  a `socat` could keep the lock open and **stall every subsequent reconciliation pass**.
  The fd is now closed at spawn (`200>&-`).

## Security

- **dockerd TCP and etcd bound to the node's private IP** — the installer no longer
  binds `dockerd:2375` or the etcd API `:2379` to `0.0.0.0` but to the node's
  **cluster IP**. 2375 is unauthenticated (root RCE) and must never listen on a public
  interface; the private bind is the real barrier, the firewall is defense-in-depth.
- **`docker_addr` validated before use** — a node's Docker endpoint (`X-Caelix-Node`
  targeting) read from the store is now validated (`tcp://host:port` or `unix://`)
  before it becomes `DOCKER_HOST`, closing an SSRF vector if the store is poisoned.
- **Hardened materialization** — writing shared certs/stacks/templates to disk rejects
  any name with a separator or `..` (anti-traversal), and compose files are written
  atomically (tmp + `rename`).
- **Cluster hardening prerequisites** (operator-side): the etcd KV is the control
  plane and holds the JWT secret, password hashes and TLS keys. In production, **enable
  TLS** on the etcd API (client and peer). Without TLS, anyone reaching the cluster's
  etcd API can read those secrets.
