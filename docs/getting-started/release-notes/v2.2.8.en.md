# Release notes — v2.2.8

**Date**: July 2026
**Image**: `ghcr.io/arcneell/caelix:2.2.8` (also `:latest`)

> Fix: **MySQL** database backups were failing silently.

---

## Fix — DB backup on MySQL images

The logical database backup (`backup_db_type = mysql|mariadb`) used `mariadb-dump` for
both engines. But **`mysql:*`** images only ship `mysqldump` (not `mariadb-dump`), so the
dump failed (`command not found`).

Before the 2.2.3 hardening this failure was **hidden** (gzip of an empty output produced
a non-empty archive → recorded as "successful"). Since 2.2.3 (`PIPESTATUS` + `gzip -t`
checks) the real failure is correctly detected — but because a failed backup doesn't
update its timestamp, it was **retried on every agent pass**, flooding incidents
(`backup_failed` in a loop).

The backup now picks the tool present in the container: `mariadb-dump` when available
(MariaDB images), otherwise `mysqldump` (MySQL images, and the MariaDB compat alias).
MySQL backups produce a valid dump again.

> If you saw looping `Backup failed` incidents on a MySQL-backed app, they stop with this
> version (the backup succeeds and records `backup_completed`).

## Validation

Verified live on the 3-node cluster: the `wordpress-db` dump (`mysql:8` image) succeeds
(~840 KB gzip, `gzip -t` OK), the incident flips from `backup_failed` to
`backup_completed`, and the retry loop stops.
