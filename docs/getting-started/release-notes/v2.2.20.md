# Notes de release — v2.2.20

**Date** : juillet 2026
**Image** : `ghcr.io/arcneell/caelix:2.2.20` (également `:latest`)

> Deux surfaces manquantes de la frontière « technicien sans secret », plus trois handlers
> etcd bloquants — trouvés par l'audit de convergence.

---

## Sécurité — le flux d'état ne fuit plus de secrets aux techniciens (HIGH)

Le flux SSE **`GET /api/stream`** renvoyait l'état **complet non assaini** à tout utilisateur
authentifié, alors que son équivalent REST `GET /api/state` applique déjà
`sanitize_state_for_role`. Un technicien recevait donc les identifiants de notification
(SMTP/webhook) et les `env`/`volumes_bind` par service que `/state` masque. Le flux applique
désormais le **même assainissement selon le rôle** du ticket SSE.

Même classe fermée sur deux autres lectures : **`GET /containers/{name}/inspect`** et
**`/clone-config`** masquent désormais `Config.Env` / `env` pour les non-admins (le `docker
inspect` brut n'est plus un contournement de la frontière).

## Backend — trois handlers etcd bloquants sortis de l'event loop

`manifest_raw_get`, `list_webhooks` (lectures) et `templates_save` (écriture) faisaient des
I/O `read_text`/`write_text` synchrones — des allers-retours etcd bloquants en mode cluster.
Passés en handler synchrone (threadpool) ou `asyncio.to_thread`.

## Validation

Suite complète (748 pytest + 306 bats + vérifications shell + build frontend) au vert, puis
déploiement `2.2.20` sur le cluster 3 nœuds live (backend-only : quorum etcd 3/3, VIP flottante,
auth etcd vérifiés).
