## Sécurité — identifiants de backup par app masqués aux techniciens (HIGH)

`GET /api/state` renvoyait, pour les sections d'app (non réservées) du manifest, **tous** les
champs sauf `env`/`volumes_bind`. Or `lib/backup.sh` lit les identifiants de sauvegarde
directement comme clés de premier niveau de la section d'app (`backup_ftp_pass`,
`backup_rsync_pass`, `backup_rsync_ssh_key`) — et aucune n'appartenait à `_NOTIFY_REDACT_KEYS`.
Un **technicien** recevait donc ces secrets en clair via `/state`, en contradiction avec la
frontière déjà appliquée à `notify` et aux sections réservées.

`sanitize_state_for_role` applique désormais un prédicat de clé-secrète
(`_is_secret_manifest_key`) à **toutes** les sections du manifest (réservées comme app) : il
couvre `_NOTIFY_REDACT_KEYS`, les noms génériques (`password`, `token`, `secret`, `api_key`,
`private_key`, `credential`) et les suffixes conventionnels (`*_pass`, `*_password`, `*_secret`,
`*_token`, `*_api_key`, `*_ssh_key`, `*_private_key`) — donc aussi les futurs champs secrets. La
frontière technicien-sans-secret est désormais close sur tous les shapes de `/state`.

## Qualité — campagne de tests unitaires (61 % → 68 %)

242 tests ajoutés sur les modules les moins couverts, tous hermétiques (Docker/etcd/SMTP/curl
monkeypatchés) :

- **Routeurs orchestrateur** : `_orch_apps` (21 % → 72 %), `_orch_notify` (17 %), `_orch_update`
  (19 %), `_orch_service`, `_orch_incidents`, `_orch_manifest` — CRUD d'apps, tests de
  notification (SMTP/Discord/Slack/Teams/Telegram), auto-update, actions de service, incidents,
  et le gating admin/technicien de chaque endpoint.
- **Cœur** : `state.py` (dont `sanitize_state_for_role` exhaustivement comme frontière de
  secrets), `certbot_runner` (33 % → 95 %, argv sûr sans interpolation shell).
- **Cluster** : `scheduler` (65 % → 100 %, placement/anti-affinité/co-location/`max_per_node`),
  `aware` (54 % → 100 %, `effective_manifest`/routing), `hpa` (80 % → 98 %, parsing CPU/replicas).
