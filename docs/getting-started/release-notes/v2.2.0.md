# Notes de release — v2.2.0

**Date** : juillet 2026  
**Image** : `ghcr.io/arcneell/caelix:2.2.0` (également `:latest`)

> Release **full cluster-aware**. Chaque chemin opérationnel — déploiement, actions
> service/conteneur, logs, vues dérivées du manifeste, sauvegardes, scale manuel,
> stacks compose, déploiement de templates, ingress, certificats Let's Encrypt et VIP
> flottante — résout désormais le nœud hôte, lit le manifeste cluster ou réplique son
> état via etcd, et **survit à une bascule de VIP**. Le mono-hôte est préservé partout.

---

## Cluster-aware

- **Certificats Let's Encrypt** répliqués sur tous les nœuds : l'ingress et le TLS
  suivent la VIP. Le proxy transmet `X-Forwarded-Proto` aux applications derrière TLS
  (corrige le CSS/JS manquant en HTTPS, ex. WordPress).
- **Actions service/conteneur, logs et vues** ciblent le nœud qui héberge réellement
  le conteneur (avant : le démon local uniquement → no-op silencieux si l'app tournait
  ailleurs).
- **Sauvegardes** exécutées sur le nœud qui détient les données ; `status`/`list`/
  `download` agrègent les archives des trois nœuds.
- **Scale manuel** via `total_replicas` dans le manifeste cluster (le contrôleur place
  les répliques sur les nœuds).
- **Stacks compose** épinglées à leur nœud de déploiement et pilotables depuis
  n'importe quel nœud ; `list` agrège l'état de tous les nœuds.
- **Bascule VIP** propre à l'arrêt gracieux de l'agent (drain via `ExecStop`) ; les
  routes de domaines et les certificats répliqués permettent au nouveau leader de tout
  servir immédiatement.

Un helper réutilisable (`core/cluster/aware.py`) centralise la résolution de nœud, le
merge du manifeste cluster et le fan-out multi-nœuds.

---

## Validation

Testé en direct sur un cluster à 3 nœuds, **bascule de VIP complète comprise** : ingress
HTTPS, action cross-nœud, statut de sauvegarde agrégé et contrôle des stacks compose
continuent tous de fonctionner sur le nouveau leader.
