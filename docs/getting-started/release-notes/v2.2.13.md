**Date** : juillet 2026
**Image** : `ghcr.io/arcneell/caelix:2.2.13` (également `:latest`)

> Correctif : un conteneur tué par manque de mémoire (OOM) n'est plus confondu avec un
> arrêt manuel de l'opérateur.

## Correctif — OOM-kill ≠ arrêt manuel

Le moteur suspend le redémarrage automatique d'un conteneur arrêté « à la main »
(`manual_stop_pause`), en se basant sur le code de sortie : `0`, `143` (SIGTERM) et
`137` (SIGKILL) sont interprétés comme un arrêt volontaire (`exit_code_looks_like_manual_stop`,
`lib/repair.sh`).

Or un conteneur **tué par l'OOM-killer sort aussi en 137**. Il était donc traité comme un
arrêt manuel → mis en pause et **jamais redémarré**, ce qui pouvait laisser un service (y
compris la console `caelix-ui`) **hors service après chaque incident mémoire**, jusqu'à
une intervention manuelle.

Docker distingue les deux via `.State.OOMKilled`. La détection d'arrêt manuel **exclut
désormais** les OOM-kills : un conteneur OOM-killé retombe sur le chemin de
redémarrage/recréation normal (comme n'importe quelle panne), tandis qu'un vrai arrêt
opérateur reste respecté.

> Découvert lors du soak test VIP (`scripts/vip_soak.sh`) : un `docker kill` du conteneur
> (exit 137) laissait la console en pause au lieu d'être relancée.

## Validation

Tests bats ajoutés (`repair_oom.bats` : 137+OOMKilled=true → redémarrage ; 137+OOMKilled=false
→ pause). Suite complète (pytest + bats + vérifications shell + build frontend) au vert,
puis déploiement de l'image `2.2.13` sur le cluster 3 nœuds live avec vérification du
quorum etcd, de la VIP flottante et de l'authentification etcd.
