# Release notes — v2.2.17

**Date**: July 2026
**Image**: `ghcr.io/arcneell/caelix:2.2.17` (also `:latest`)

> Targeted cluster fix: a removed node can no longer **resurrect** itself.

---

## Cluster — removal tombstone (H3)

`remove_node` deleted a node's keys but wrote **no tombstone**. A partitioned-but-alive node
(or one removed while healthy) **re-published** its keys (`meta`/`heartbeat`/backends) on its
next agent cycle and **reappeared** in the cluster — the scheduler re-targeted it, ingress
re-pointed at it → double-run and flapping. The self-fence the docs promised did not exist.

Now:
- `remove_node` writes a **leased tombstone** `caelix/nodes/<id>/tombstone` (after deleting the
  node's keys in **one atomic `deleterange`** — also fixes the earlier non-atomicity);
- the agent reads the tombstone at the top of `node_agent_cycle`: if present → **self-fence**,
  release the VIP, publish nothing;
- an **authorized re-join** (gated by the single-use join token) clears the tombstone, so a
  removed node can be cleanly re-added; the lease also expires the tombstone as a safety net.

## Validation

Tests: remove → tombstone present + keys gone + re-join clears (FakeEtcd); agent self-fences
under a tombstone / proceeds normally otherwise (bats). Full suite (747 pytest + 306 bats +
shell checks + frontend build) green, then `2.2.17` deployed to the live 3-node cluster (etcd
3/3 quorum, floating VIP, etcd auth verified).
