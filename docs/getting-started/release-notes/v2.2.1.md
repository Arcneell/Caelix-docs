# Notes de release — v2.2.1

**Date** : juillet 2026
**Image** : `ghcr.io/arcneell/caelix:2.2.1` (également `:latest`)

> Release **durcissement sécurité**. Audit complet du code (auth, exécution Docker,
> plan de contrôle cluster, secrets, scripts d'installation) suivi de la correction de
> l'ensemble des findings. Le comportement fonctionnel est inchangé ; les nouveautés
> sécurité côté cluster sont **rétro-compatibles** (activées seulement si configurées).

---

## Plan de contrôle cluster (etcd)

- **Authentification client etcd** : la console et l'agent s'authentifient auprès
  d'etcd (`/v3/auth/authenticate`, jeton rafraîchi sur 401). L'installeur active l'auth
  de façon idempotente au bootstrap (par défaut en mode cluster ; `--no-etcd-auth` pour
  s'en passer). La simple accessibilité réseau de `:2379` ne suffit plus à lire le KV.
- **Chiffrement des secrets au repos** : le secret de signature JWT, les comptes
  utilisateurs et les clés privées TLS sont chiffrés (Fernet/AES) avant stockage dans
  etcd. La clé maîtresse (`CAELIX_MASTER_KEY`) n'est **jamais** stockée dans etcd — elle
  est distribuée à chaque nœud via le fichier d'environnement cluster (0600). Le
  plaintext hérité est lu de façon transparente puis re-chiffré (migration au boot).
- **Jetons de join à usage unique** : `Cluster → Ajouter un nœud` émet un jeton borné
  dans le temps, consommé de façon atomique au premier usage — une commande de join ne
  peut plus être rejouée.

## Auth & contrôle d'accès

- Comparaison à temps constant du jeton hérité `CAELIX_UI_TOKEN`.
- Le rôle est relu depuis la base à chaque requête (une rétrogradation/désactivation
  prend effet immédiatement au lieu d'attendre l'expiration du JWT).
- Verrouillage de connexion **par identifiant** (plafond global) en plus du par-IP :
  un attaquant qui alterne les IP ne peut plus forcer un compte.
- `/metrics` requiert l'authentification **par défaut** (`CAELIX_METRICS_PROTECT=0`
  pour l'exposer à un scrape Prometheus interne sur réseau de confiance).
- `exec` interdit dans les conteneurs système Caelix (console/etcd) + journalisé.

## Exécution Docker & ingress

- `docker run` (création de conteneur) rejette les valeurs d'option préfixées d'un
  tiret (injection d'arguments) et les chemins `env_file` avec traversée.
- L'autoscaler (HPA) valide `docker_addr` avant d'en faire un `DOCKER_HOST` (fermeture
  d'un vecteur SSRF → RCE côté contrôleur).
- Les backends d'ingress lus du registre sont validés (`host:port`) avant d'alimenter
  le proxy — une entrée empoisonnée est écartée.

## Secrets & scripts

- Fichiers de secrets (secret JWT, clés TLS, `notify.ini`) écrits en 0600 **dès la
  création** (plus de fenêtre lisible).
- `notify.ini` / `manifest.ini` mis en 0600 par l'installeur (mots de passe SMTP,
  jetons de bot, webhooks, identifiants de sauvegarde).
- Identifiants SMTP/FTP passés via un fichier `curl --config` 0600 au lieu de la ligne
  de commande (l'argv est lisible dans `/proc`).
- Validation des clés SSH/known_hosts rsync (blocage de l'injection d'options
  `ssh -o ProxyCommand=…`).
